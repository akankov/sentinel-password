# Internationalization (i18n)

`@sentinel-password/core` does not have built-in i18n yet. Validators emit short, stable English strings; pluggable message templates are on the roadmap. Until then, you can localize at the application layer with a simple lookup table.

## What gets emitted in English

`validatePassword` collects each failing validator's `message` into `result.feedback.suggestions` (a `readonly string[]`) and surfaces the first one as `result.feedback.warning`. There are three categories of messages — the translation strategy depends on which one you're dealing with:

| Category | Validators | Shape |
|----------|------------|-------|
| **Static** | `sequential`, `keyboardPattern`, `commonPassword`, `personalInfo` | One fixed English string per validator (no placeholders) — safe to use literally as a translation key |
| **Number-dynamic** | `length` (`minLength` + `maxLength`), `repetition` (`maxRepeatedChars`) | English template with a number baked in. The template wording is stable, the number varies per policy. |
| **List-dynamic** | `characterTypes` | English template with a comma-joined list baked in. The order is stable (`uppercase letter, lowercase letter, digit, symbol`) but the subset varies — up to 15 combinations. |

The English wording is stable across patch and minor releases, so all three categories can be matched against — see strategies below.

## Translation Pattern

Map each English suggestion to your locale with a `Record<string, string>` lookup. Fall back to the original string if a key is missing — that way new validator messages don't crash the UI when you haven't translated them yet.

```typescript
import { validatePassword } from '@sentinel-password/core'

const translations: Record<string, Record<string, string>> = {
  es: {
    'Password must be at least 8 characters': 'La contraseña debe tener al menos 8 caracteres',
    'Password is too common. Please choose a more unique password.':
      'La contraseña es demasiado común. Elija una contraseña más única.',
    'Password contains sequential characters (e.g., abc, 123)':
      'La contraseña contiene caracteres secuenciales (por ejemplo, abc, 123)',
    'Password contains common keyboard patterns':
      'La contraseña contiene patrones de teclado comunes',
    'Password contains personal information': 'La contraseña contiene información personal',
  },
  fr: {
    'Password must be at least 8 characters':
      'Le mot de passe doit comporter au moins 8 caractères',
    // ...
  },
}

function localize(messages: readonly string[], locale: string): string[] {
  return messages.map((msg) => translations[locale]?.[msg] ?? msg)
}

const result = validatePassword(password, options)
const localized = localize(result.feedback.suggestions, 'es')
```

## Number-dynamic messages

Three messages bake a configured number into the English string:

- `Password must be at least {minLength} characters`
- `Password must be at most {maxLength} characters`
- `Password contains too many repeated characters (max {maxRepeatedChars})`

Two strategies:

**Option A — translate each key you actually use.** If your policy is `minLength: 12`, `maxLength: 128`, `maxRepeatedChars: 3`, you only need translations for the three literal strings those values produce. Pragmatic for small, fixed policy sets — and remember to include `maxLength` even though users rarely hit it (the default ceiling is 128 characters), otherwise edge-case rejections fall back to English.

**Option B — parse and re-template.** Match each English template with a regex, capture the number, and re-format into the locale's template. Covers every value without per-value keys.

```typescript
const numericPatterns: Record<string, Array<[RegExp, string]>> = {
  es: [
    [
      /^Password must be at least (\d+) characters$/,
      'La contraseña debe tener al menos {n} caracteres',
    ],
    [
      /^Password must be at most (\d+) characters$/,
      'La contraseña debe tener como máximo {n} caracteres',
    ],
    [
      /^Password contains too many repeated characters \(max (\d+)\)$/,
      'La contraseña tiene demasiados caracteres repetidos (máx. {n})',
    ],
  ],
}

function localizeNumeric(msg: string, locale: string): string | null {
  for (const [pattern, template] of numericPatterns[locale] ?? []) {
    const match = msg.match(pattern)
    if (match) return template.replace('{n}', match[1]!)
  }
  return null
}
```

## List-dynamic message (`characterTypes`)

The character-types validator joins the missing type names with `, ` in a fixed order (`uppercase letter, lowercase letter, digit, symbol`). So with `requireUppercase: true, requireDigit: true, requireSymbol: true` on a password missing all three, the message is:

```
Password must contain at least one uppercase letter, digit, symbol
```

That's one of up to 15 combinations (every non-empty subset of the four require-types). Translating each combination literally doesn't scale; better to parse out the list and translate the **type names** individually, then re-join with the locale's list separator.

```typescript
const typeNames: Record<string, Record<string, string>> = {
  es: {
    'uppercase letter': 'letra mayúscula',
    'lowercase letter': 'letra minúscula',
    digit: 'dígito',
    symbol: 'símbolo',
  },
}

const charTypesPrefixes: Record<string, string> = {
  es: 'La contraseña debe contener al menos un',
}

function localizeCharTypes(msg: string, locale: string): string | null {
  const match = msg.match(/^Password must contain at least one (.+)$/)
  if (!match) return null
  const types = match[1]!.split(', ')
  const names = typeNames[locale]
  const prefix = charTypesPrefixes[locale]
  if (!names || !prefix) return null
  return `${prefix} ${types.map((t) => names[t] ?? t).join(', ')}`
}
```

The English type names (`uppercase letter`, `lowercase letter`, `digit`, `symbol`) are stable across releases — they live in [`character-types.ts`](https://github.com/akankov/sentinel-password/blob/main/packages/core/src/validators/character-types.ts).

## Roadmap

A future release will expose pluggable message templates so you can pass a `messages` option (or a translator function) to `validatePassword` and skip the lookup-table step entirely. Until then, the lookup-table pattern above is the recommended approach.
